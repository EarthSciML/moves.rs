# Phase 1 coverage map

Phase 1 Task 9 (bead `mo-55l0`) deliverable. Where the per-fixture
`execution-trace.json` files under `characterization/snapshots/`
answer "which MOVES code was exercised by *this* fixture?", the
`coverage-map.json` here answers the same question rolled up across
the whole fixture suite:

* For each MOVES Java class, SQL macro file, and Go calculator —
  how many fixtures invoke it, how many worker bundles reference it,
  and what fraction of the suite-wide SQL-statement workload runs
  through code paths that touch it.
* A flat, cross-category `hot_paths` list — the top-N items by combined
  fixture coverage and statement-weight share. This is the migration-
  ordering input for Phase 3: port the hottest things first so the
  Rust binary's regression-against-canonical-MOVES surface grows as
  fast as possible.

## Files in this directory

| File | Purpose |
|------|---------|
| `coverage-map.json` | Suite-wide rollup. Schema: `moves-coverage/v1`. Byte-deterministic for a fixed set of input traces. Regenerated by `moves-coverage`. |
| `README.md`         | This file. |

## Regenerating the coverage map

```sh
cargo run --release --bin moves-coverage -- \
    --snapshots-dir characterization/snapshots \
    --output-dir   characterization/coverage
```

The binary walks `characterization/snapshots/<fixture>/` for every
subdirectory that contains an `execution-trace.json`. Subdirectories
without a trace file are silently skipped, so the tool runs cleanly
even on a partially populated snapshots directory (e.g. while a fixture
suite is in flight).

The output is byte-deterministic given the same set of input traces:
re-running with no changes produces an identical `coverage-map.json`.

## Schema (`moves-coverage/v1`)

Top-level fields:

| Field | Type | Description |
|-------|------|-------------|
| `coverage_version`       | string | Schema tag. Bumped on incompatible changes. |
| `total_fixtures`         | int    | Number of `execution-trace.json` files aggregated. Denominator for `*.fixture_fraction`. |
| `total_statement_weight` | int    | Suite-wide sum of `worker_bundles[].statement_count`. Denominator for `*.statement_weight_fraction`. |
| `fixtures`               | array  | Per-fixture summaries (`fixture_name`, `sif_sha256`, `runspec_sha256`, counts) sorted by name. |
| `java_classes`           | array  | Per-class coverage rows, sorted by descending `score` with `name` as tiebreaker. |
| `sql_files`              | array  | Per-SQL-macro coverage rows, sorted by descending `score` with `id` as tiebreaker. |
| `go_calculators`         | array  | Per-Go-calculator coverage rows, sorted by descending `score` with `id` as tiebreaker. |
| `hot_paths`              | array  | Flat cross-category ranking, capped at 50 entries. |

Each coverage row carries:

| Field | Type | Description |
|-------|------|-------------|
| `name` / `id`            | string | Java FQN, SQL path, or Go calc name. |
| `kind`                   | string | (Java only.) `calculator`, `generator`, `framework`, `worker`, `master`, `common`, `utils`, `other`. |
| `fixture_count`          | int    | Distinct fixtures whose trace contains this item. |
| `fixture_fraction`       | float  | `fixture_count / total_fixtures`. |
| `fixtures`               | array  | Fixture names where this item appears, sorted alphabetically. |
| `bundle_count`           | int    | `(fixture, worker_bundle)` pairs referencing the item in `worker.sql`. Class-load-log-only Java classes get 0 here. |
| `statement_weight`       | int    | Sum of `bundle.statement_count` over the bundles in `bundle_count`. |
| `statement_weight_fraction` | float | `statement_weight / total_statement_weight`. |
| `score`                  | float  | `(fixture_fraction + statement_weight_fraction) / 2`. Used for ranking. |

`hot_paths` entries carry `kind` (`java_class` / `sql_file` /
`go_calculator`), `id`, `fixture_count`, `statement_weight`, and the
same `score` as the per-category row.

## How the weighting proxy works

`ExecutionTrace` does not carry timing data — the Phase 0 instrumentation
captures *which* MOVES code ran but not *for how long*. The coverage
map uses `WorkerBundle::statement_count` (non-empty, non-comment lines
in the bundle's `worker.sql`) as the proxy:

* For every Java class / SQL file / Go calc that appears in a bundle's
  per-bundle list, the bundle's full `statement_count` is added to that
  item's `statement_weight`.
* `total_statement_weight` is the suite-wide sum of `statement_count`,
  so each item's `statement_weight_fraction` is in `[0, 1]`.

This is a coarse approximation — a class referenced once in a 10k-statement
bundle gets the same weight as one referenced 50 times in the same
bundle. The resulting ranking is meant to drive migration-ordering
heuristics ("port the things that show up in big, common bundles
first"), not precise time accounting. If Phase 3 needs sharper
attribution later, the trace builder is the right place to add
per-class statement counts.

## Inspecting the coverage map

```sh
# Suite-wide totals:
jq '{total_fixtures, total_statement_weight}' \
    characterization/coverage/coverage-map.json

# Top-10 migration-priority items across all categories:
jq '.hot_paths[0:10] | map({kind, id, score})' \
    characterization/coverage/coverage-map.json

# Calculators only:
jq '.java_classes | map(select(.kind == "calculator"))[0:10] |
    map({name, fixture_count, statement_weight, score})' \
    characterization/coverage/coverage-map.json

# How is `BaseRateCalculator.sql` exercised?
jq '.sql_files[] | select(.id | endswith("BaseRateCalculator.sql"))' \
    characterization/coverage/coverage-map.json
```

## State of the coverage map in this commit

The map shipped here is generated against whatever execution traces
exist under `characterization/snapshots/`. When the upstream Phase 0
fixture-capture pipeline (the patched-MOVES SIF + `run-all-fixtures.sh`)
has not yet been run in a given checkout, the snapshot subdirectories
contain no traces and the coverage map is structurally valid but
empty (`total_fixtures = 0`, all collections empty). That is the
expected initial state — `moves-coverage` is the deliverable; the
populated map follows the first end-to-end fixture-suite capture.

## Phase 3 task ordering — the consumer side

Phase 3 ports MOVES calculators into Rust in coverage-map order. The
migration-plan task numbering (`Task 28` onward) is set up so the
calculator with the highest `hot_paths` score lands first, and so on
down the list, with priority ties resolved by source-line size. The
expected top entries — given the canonical-MOVES fixture suite shipped
in Phase 0 — are:

* `database/CalculatorSQL/BaseRateCalculator.sql` (~1.6k lines; the
  foundation calculator every chain depends on).
* `database/CalculatorSQL/CriteriaRunningCalculator.sql` (~1.2k lines).
* `database/CalculatorSQL/HCSpeciationCalculator.sql` (~1.0k lines).
* `database/CalculatorSQL/ActivityCalculator.sql` (~1.0k lines).
* `BaseRateGenerator.go` (~2.4k lines; primary Go-sourced calculator).
* `OperatingModeDistributionGenerator`, `TotalActivityGenerator`
  (Java-sourced generators that fire under every running/criteria run).

The first populated coverage map should rank these at the top of
`hot_paths`. If it doesn't, that's a signal the instrumentation, the
trace builder, or the fixture-suite RunSpec mix is missing something —
file a bead, don't reorder Phase 3 silently.
