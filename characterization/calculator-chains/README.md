# Phase 1 calculator-chain DAG

Phase 1 Task 10 (bead `mo-78un`) deliverable. The companion to
`characterization/coverage/coverage-map.json`: where the coverage map
answers *"which MOVES code did the fixture suite exercise?"*,
`calculator-dag.json` here answers *"how do those calculators interconnect,
and in what order do they fire?"*.

The DAG is reconstructed from three sources:

1. `CalculatorInfo.txt` — the runtime log MOVES writes via
   `gov.epa.otaq.moves.master.framework.InterconnectionTracker` when the
   build flag `CompilationFlags.GENERATE_CALCULATOR_INFO_DOCUMENTATION` is
   on. The file in the pinned MOVES tree contains **1,246** directives
   (1,134 `Registration` + 83 `Subscribe` + 29 `Chain`), matching the
   migration plan's headline number.
2. The MOVES Java source tree at the same pinned commit
   (`25dc6c833dd8c88198f82cee93ca30be1456df8b` —
   MOVES5.0.1 release tip on master, 2025-11-14). The reconstructor scans
   `*.java` files under `gov/epa/otaq/moves/master/` and fills in
   granularity / priority for direct-subscriber calculators that didn't
   fire during the run that produced `CalculatorInfo.txt` (typically
   because no fixture in the suite exercised them).
3. The `MasterLoopGranularity` and `MasterLoopPriority` enums, encoded
   into the `moves-calculator-info` crate so the JSON can use the
   canonical text forms (`"MONTH"`, `"EMISSION_CALCULATOR+1"`) instead of
   re-deriving numeric ordering at every consumer.

## Regenerating the DAG

```sh
# Pin the MOVES source to the same commit
# characterization/apptainer/files/versions.env points at:
git clone https://github.com/USEPA/EPA_MOVES_Model.git /tmp/moves-source
git -C /tmp/moves-source checkout 25dc6c833dd8c88198f82cee93ca30be1456df8b

cargo run --release -p moves-calculator-info -- \
    --calculator-info /tmp/moves-source/CalculatorInfo.txt \
    --source-dir      /tmp/moves-source \
    --output-dir      characterization/calculator-chains
```

The binary walks every `*.java` under `--source-dir`, recovers
subscription metadata for the calculators it recognises, builds the DAG
out of `CalculatorInfo.txt`'s directives, and writes
`calculator-dag.json`.

The output is byte-deterministic given the same inputs:

```sh
cargo run --release -p moves-calculator-info -- ... # run 1
sha256sum characterization/calculator-chains/calculator-dag.json
cargo run --release -p moves-calculator-info -- ... # run 2
sha256sum characterization/calculator-chains/calculator-dag.json
# identical hashes
```

### How Phase 2 uses this

The `CalculatorRegistry` (Phase 2 Task 19, bead `mo-uhq9` →
`crates/moves-framework`) loads `calculator-dag.json` at startup:

1. For every entry in `modules` it instantiates the corresponding Rust
   struct via a static lookup table (port one-to-one with MOVES's
   `MOVESInstantiator.java`).
2. For every direct subscriber it hooks the calculator into the Rust
   port's MasterLoop equivalent at the declared `(granularity,
   priority)`.
3. At RunSpec resolution time, given a `(process_id, pollutant_id)` the
   RunSpec asks for, the registry looks the entry up in
   `execution_chains` and runs the listed `roots` (and their
   `chain_templates`' transitive `steps`) in order.

The `chain_templates` table is intentionally **deduplicated by root**:
the downstream chain a root drags along is the same regardless of which
`(process, pollutant)` triggered it (Java semantics —
`EmissionCalculator.chainedCalculators` is flattened from the root's
perspective). Listing it once per root keeps the JSON document under
1 MB even when 960 `(process, pollutant)` pairs reference it.

### Granularity ordering

Java's
[`MasterLoopGranularity.compareTo`](https://github.com/USEPA/EPA_MOVES_Model/blob/HEAD/gov/epa/otaq/moves/master/framework/MasterLoopGranularity.java)
flips the natural integer order so that the MasterLoop iterates
**coarse-first** (PROCESS → STATE → COUNTY → ZONE → LINK → YEAR → MONTH
→ DAY → HOUR). `execution_index.granularity_index` bakes that flip in:
**smaller indices fire earlier**. `MATCH_FINEST` is a special sentinel
that pins a calculator to the finest granularity in play and therefore
sorts last (largest `execution_index`).

### Priority ordering

Java's
[`MasterLoopPriority`](https://github.com/USEPA/EPA_MOVES_Model/blob/HEAD/gov/epa/otaq/moves/master/framework/MasterLoopPriority.java)
defines three named tiers with integer offsets:

| Constant                    | Value | Notes                                    |
|-----------------------------|-------|------------------------------------------|
| `INTERNAL_CONTROL_STRATEGY` | 1000  | AVFT / RoP / Retrofit override calcs     |
| `GENERATOR`                 |  100  | Activity, OpMode, FuelEffects, …         |
| `EMISSION_CALCULATOR`       |   10  | The actual emission-producing calcs      |

Higher values fire **earlier** within the same granularity bucket. Bands
of ±5 (calculator), ±50 (generator), ±500 (strategy) around each constant
get rendered as `"NAME+N"` / `"NAME-N"` in the JSON; anything outside
the bands is the raw integer literal.

## Caveats

* `CalculatorInfo.txt` is a runtime log — only what fired during the
  particular MOVES run that produced it appears. The pinned source ships
  one such log, but Java-source scanning fills in subscriptions for
  calculators that never fired in that run (e.g.
  `BasicBrakeWearPMEmissionCalculator` at YEAR granularity).
* Per the Java implementation
  ([`AirToxicsCalculator`](https://github.com/USEPA/EPA_MOVES_Model/blob/HEAD/gov/epa/otaq/moves/master/implementation/ghg/AirToxicsCalculator.java)
  is a representative example), chained calculators determine their
  upstream by querying `EmissionCalculatorRegistration.find(pollutant,
  process)` at subscribe time — i.e. the chain target depends on
  which calculators registered for the input pollutant. The `Chain`
  directives in `CalculatorInfo.txt` are a flat union of every such
  hookup across pollutants; Phase 2 RunSpec resolution narrows them per
  `(process, pollutant)`.
* `MeteorologyGenerator` and a handful of other generators subscribe at
  `PROCESS` granularity. They fire once per process, **before** anything
  inside `STATE / COUNTY / ... / HOUR` for that process — the
  `global_execution_order` view captures this.
